## HIGH ‚Äî leaderboard fairness (surgical fixes)

### 1. Quiz submissions accepted after the window closes
`src/app/dashboard/events/actions.ts` ‚Äî `submitAttemptAction` fetches
`status/startsAt/endsAt` but never checks them; only double-submit is refused.
A tab opened before `endsAt` (or before staff close the event) can submit
hours later and land on the leaderboard. Contradicts the code's own contract
("the server is the source of truth", quiz.ts:173, TakeClient.tsx:12).
**Fix:** refuse when `!isEventOpen(...)` (small grace window for in-flight
auto-submits).

### 2. LIVE events takeable as self-paced quizzes via `/take` (cheat vector)
`take/page.tsx` never checks `event.mode` and `isEventOpen` passes for status
LIVE, so during a live session a student opens `/take` in a second tab, sees
the entire frozen question set, copies answers as the host reveals them, and
`submitAttemptAction` (also no mode check) **overwrites their live attempt**
with a near-perfect score. The events list links every available event to
`/take` under a stale "the take flow will gate them" comment.
**Fix:** redirect students to `/play` when mode is LIVE; mode-aware list link;
refuse LIVE events in `submitAttemptAction`.

### 3. Weekly-challenge submissions accepted after `closedAt`
`src/app/dashboard/challenges/actions.ts` ‚Äî `submitWeeklyChallengeAction`
selects `closedAt` but never compares it to now. Also
`challenges/[id]/attempt/page.tsx:53` checks the **current week's** window,
not the loaded challenge's own `openedAt/closedAt`, so a stale week-N attempt
can be reopened and submitted during any later live Saturday. Ties break on
`submittedAt asc`, so researched late submits outrank honest ones.
**Fix:** window-check the challenge's own bounds in both action and page.

## HIGH ‚Äî billing & communications integrity (need design decisions)

### 4. Checkout clobbers the live subscription before payment
`billing/actions.ts:162-174` ‚Äî the single Subscription row is upserted to
`{planId: newPlan, status: 'trialing', razorpaySubId: newOrderId}` the moment
Subscribe is clicked. Abandon the Razorpay modal ‚Üí a paying Pro school is
silently dropped to Free limits, and the billing page's `isCurrent` (slug-only)
disables the retry button. Variant: clicking plan A then plan B orphans
order A ‚Äî paying it charges the school with no activation (webhook `updateMany`
matches nothing).
**Fix:** persist the pending order separately (pending columns / table); flip
plan+status only in the webhook on `order.paid` (it already has tenant/plan in
order `notes`); make `isCurrent` status-aware.

### 5. One payment buys a "monthly" plan forever
`api/webhooks/razorpay/route.ts` ‚Äî checkout creates a one-time ORDER;
`order.paid` sets `status='active'` but `currentPeriodEnd` is only extracted
from subscription entities (stays null), nothing ever compares it to now, and
there is no billing cron. `priceMonthly` and the "Renews" label are fiction.
**Fix:** set `currentPeriodEnd = paidAt + 1 month` on order.paid; treat
`active` + elapsed period as lapsed in `getTenantPlanLimits` (or a cron flips
to `past_due`).

### 6. Campaign send: stuck SENDING + double-send on retry
`communications/actions.ts` ‚Äî up to 500 sequential ZeptoMail calls in one
action; `sendEmail`'s fetch has no timeout; recipient rows are written only
AFTER the loop. A mid-loop function timeout records nothing, leaves the
campaign in SENDING forever (no recovery path), and the natural re-send
duplicates emails to everyone already delivered.
**Fix:** create PENDING recipient rows up front; mark SENT/FAILED per row;
`AbortSignal.timeout` on the fetch; `export const maxDuration` or move sending
to a cron-drained queue keyed on PENDING rows (idempotent retries).

## MEDIUM

7. **Per-attempt time limit is cosmetic** ‚Äî TakeClient starts the countdown
   from `Date.now()` at mount (reload = full reset) and the server never
   compares against the persisted `QuizAttempt.startedAt`.
   *Fix:* compute remaining from `startedAt`; reject `now > startedAt + limit + grace` server-side.
8. **LIVE MSQ with one correct option scores 0 while the reveal says
   "Correct"** ‚Äî PlayClient serializes one tick as `'"a"'` (MCQ shape); the
   server grades MSQ expecting an array. *Fix:* normalize payload shape
   server-side in `submitLiveAnswerAction` (it knows the type).
9. **The "frozen" question set isn't** ‚Äî editing/deleting a question
   referenced by a published event or live challenge silently splits grading
   keys between early/late submitters, inflates result percentages (3/2 =
   150% is reachable), desyncs LIVE host vs student indexes, and makes the
   LEGEND badge unreachable mid-challenge. The edit page's "Changes apply to
   future quiz events" copy is false. *Fix:* block edit/delete (or require
   deactivate-instead) when referenced by a non-draft, non-closed event or an
   open challenge; longer-term snapshot question content at publish.
10. **LIVE students who disconnect vanish from final results** ‚Äî
    `submittedAt` is only stamped by the student's own browser observing
    phase ENDED; close the tab early and the attempt is excluded (results
    filter `submittedAt != null`) and never badged. *Fix:* updateMany at
    end-of-event server-side.
11. **Bulk question import**: one sequential INSERT per row (timeout = silent
    partial import + duplicate-on-retry) and the 500-row cap is client-side
    only (any TEACHER can post unbounded rows). *Fix:* enforce cap server-side;
    validate-all-then-`createMany` in chunks.
12. **`imageUrl` is authored but never rendered anywhere** ‚Äî figure-based
    questions are unanswerable; the value also accepts any scheme
    (`javascript:`/`data:` persisted today = latent XSS sink for whenever
    rendering is added). *Fix:* render on take/play/practice; accept only
    http(s) via `new URL()`.
13. **Webhook has no idempotency/ordering guard** ‚Äî a redelivered
    `payment.captured` re-activates a subscription the super-admin canceled;
    out-of-order `subscription.charged` overwrites `past_due`. *Fix:* record
    event id/created_at, skip seen/older; clear `razorpaySubId` on cancel.
14. **DeleteSchoolDialog isn't a real modal** ‚Äî `aria-modal` declared but no
    Escape handling, no focus trap, no initial focus, no focus restore, no
    scroll lock. *Fix:* one `useEffect` keyed on `open`.
15. **OIDC `verifyAccessToken` validates issuer but not audience or `typ`** ‚Äî
    sharper than it looks: authorization *codes* are JWTs from the same key,
    so an intercepted code (the thing PKCE defends against) passes as a bearer
    token at `/api/oidc/userinfo` and `/api/tenant/profile`, partially
    bypassing PKCE (bounded by the short code TTL; data limited to the token's
    own claims + tenant brand). *Fix:* require `typ: 'at+jwt'` and validate
    `aud` against registered client ids in `verifyAccessToken`.

## LOW

16. **Double-submit guard is check-then-act** ‚Äî replace the final update with
    `updateMany({ where: { id, submittedAt: null } })`, treat count 0 as
    already-submitted.
17. **`startLiveAction` can restart a quiz mid-game** ‚Äî guards status but not
    `livePhase !== 'LOBBY'`; a stale host tab rewinds the class to Q1.
18. **"Close it to make changes" error is wrong** ‚Äî closing never enables
    editing and there's no CLOSED‚ÜíDRAFT path; `closeEventAction` also accepts
    DRAFT events (one-way trap; only escape is Delete, which destroys
    attempts). Fix message; restrict close to SCHEDULED/LIVE.
19. **`?sort=toString` crashes the question bank** ‚Äî `sort in SORT_ORDER`
    matches `Object.prototype` keys; Prisma throws on the function value. Use
    `Object.hasOwn`.
20. **Practice Zone ships `correctIds` for the same active pool open events
    draw from** ‚Äî a student can drill the subject during an open event and
    learn the answers. Exclude ids frozen into open events, or accept and
    document the trade-off.
21. **Campaign subject (and `schoolName`) interpolated unescaped into email
    HTML** ‚Äî tenant admin can embed arbitrary markup/links in platform-branded
    email. Escape the subject like the body.
22. **Sunday challenge card shows yesterday's date** as "The next one opens" ‚Äî
    `getChallengeWindow` returns the current week's (past) Saturday on
    Sundays. Show `openedAt + 7d`.
23. **Plan limits are TOCTOU** ‚Äî concurrent creates/imports can exceed
    event/student caps (bounded overshoot; caps are the paywall). Re-verify in
    a transaction or document as advisory.
24. **`trialEndsAt` is written but never read** ‚Äî 14-day trials never end;
    TRIAL behaves exactly like ACTIVE forever. Enforce in `withTenant` (or a
    cron), or drop the field.
25. **Staff preview can fire the real submit** ‚Äî one-SHORT-question events
    submit on Enter (implicit submission; preview hides the button but the
    form keeps the real action), bouncing the previewing teacher to /login via
    `requireRole('STUDENT')`. Guard `if (preview) preventDefault` +
    `<fieldset disabled>`.
26. **Drag-over highlight flickers** in both bulk-import drop zones
    (dragenter/dragleave child events). Use a depth counter or
    `relatedTarget` check.
27. **Password reset isn't tenant-bound** *(hardening only ‚Äî downgraded from a
    reviewer's "critical": the 32-byte token IS the credential and only ever
    resets its own user, so a different subdomain grants nothing extra; the
    unscoped design is documented in the schema)*. Optionally verify the
    token's user belongs to the requesting tenant for defense-in-depth.

## Suggestions

- **No security headers configured** (CSP, HSTS, X-Frame-Options) ‚Äî add via
  `headers()` in next.config.ts; at minimum HSTS + frame-ancestors.
- `sendEmail` should always use `AbortSignal.timeout` (affects welcome/reset
  mails too, not just campaigns).
- The bulk-import duplicate-on-retry note (#11) could be softened with a
  content-hash dedupe per tenant.
